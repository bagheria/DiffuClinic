3. Technical Issues — Version Conflicts & Library Incompatibilities

The central challenge: LLaDA and LLaMA require incompatible dependency versions. LLaDA models were released alongside transformers 4.38.2 and use APIs that were removed in later versions. LLaMA 3.1 requires transformers ≥ 4.46.0 to handle rope_scaling configs. These cannot coexist.

3.1 Dependency Version Matrix

PackageLLaDA EnvironmentLLaMA EnvironmentConflict?
transformers 4.38.2 4.46.0 Yes — core incompatibility
peft 0.10.0 0.19.1 Yes — LoRA adapter format differs
flash-attn Not used 2.8.3 No
hf-xet Must be removed in both Yes — disk quota issue
accelerate Any recent version No
Resolution strategy: Used separate conda environments per architecture. llada_env with transformers==4.38.2 + peft==0.10.0 for LLaDA models; llama_env with transformers==4.46.0 + peft==0.19.1 + flash-attn for LLaMA models.

3.2 All Issues Encountered

Category A — Version / API Incompatibilities
#1

peft / transformers version conflict

Latest peft requires newer transformers APIs not present in 4.38.2. Installing peft without pinning pulls incompatible transformers.
pip install peft==0.10.0 — pin to the version compatible with transformers 4.38.2
#2

rope_scaling ValueError on LLaMA 3.1

Llama 3.1 uses "rope_type": "llama3" in its config. Transformers 4.38.2 has no handler for this key.
Upgrade to transformers==4.46.0 in the LLaMA environment
#3

GenerationMixin import error

transformers 5.9.0 (latest at the time) restructured internal modules. GenerationMixin path changed, breaking AutoModelForCausalLM.
Downgrade from 5.9.0 to transformers==4.46.0 — the last stable 4.x release
#4

LoraConfig.alora_invocation_tokens TypeError

The clinical LoRA adapter was trained with a newer peft version that introduced alora_invocation_tokens. peft 0.10.0 does not recognize this config key.
pip install peft==0.19.1 in the LLaMA environment
Category B — Infrastructure / RunPod
#5

xet download "Disk quota exceeded"

RunPod's /workspace uses a network filesystem with per-user quota limits. The hf-xet package writes temporary files that exceed the quota during large model downloads.
pip uninstall hf-xet — forces HuggingFace Hub to fall back to HTTP download, which respects partial writes better
#6

Root disk full — HF cache on 20GB overlay

RunPod's root filesystem is a 20GB overlay on top of a network volume. HuggingFace caches models under ~/.cache/huggingface/ by default, consuming the limited overlay.
export HF_HOME=/workspace/cache — redirect cache to the larger network mount
#7

SSH kills process on disconnect

Running python inference_llada.py as a foreground process. When the SSH session drops, the shell sends SIGHUP to all child processes.
nohup python inference_llada.py ... > run.log 2>&1 & — decouple from terminal
#8

SafetensorError: incomplete file

Model download corrupted when the xet quota was hit mid-transfer. The safetensors file was partially written but the hub marked it as complete.
Delete the corrupted model directory under HF cache, uninstall hf-xet, re-download
Category C — Code-level Bugs
#9

dLLMCache.cache_type AttributeError

Called dLLMCache.reset_cache() on the class instead of an instance. The class has no cache_type attribute — only instances do.
Create instance: dLLMCache() → call .reset_cache() on it. Wrapped in a mutable list for graceful fallback on failure.
#10

Checkpoint sorting bug — 900 > 1800

Python's default string sort puts "checkpoint_900" after "checkpoint_1800" because '9' > '1' lexicographically. Resume picked up stale checkpoint 900 instead of 1800.
Numeric sort key: sorted(files, key=lambda f: int(f.split('_')[1].split('.')[0]))
#11

LLaMA gated model — HTTP 401

meta-llama/Llama-3.1-8B-Instruct is a gated HuggingFace repo requiring authenticated access. No HF_TOKEN was set in the environment.
export HF_TOKEN=<token> and run huggingface-cli login

4. Architecture Comparison — Acceleration & Fairness

Diffusion (LLaDA)Autoregressive (LLaMA)
Generation paradigm Iterative denoising (512 steps, block-wise) Left-to-right token prediction (~127 tokens)
Accelerator dLLM-Cache Flash Attention 2
Bottleneck addressed Compute: 512 full forward passes per sample Memory: large attention matrices at long sequences
Speedup on this task ~5× (critical — without it ~15s/sample) <1% (negligible — sequences too short)
Why this is fair Each architecture uses its best available acceleration method. Comparing bare-metal LLaDA (15s/sample) vs Flash-Attention LLaMA (3.3s) would be misleading. The fair comparison is: optimal diffusion vs optimal AR, both at ~3s/sample.
dLLM-Cache mechanism: Caches transformer block activations across adjacent diffusion steps. Only re-computes blocks where the token assignment changed (transfer_ratio=0.25 — ~25% of blocks per step). Since adjacent diffusion steps are highly correlated, most blocks can reuse cached outputs.

Flash Attention 2 mechanism: Fuses QKV attention operations in GPU SRAM using tiling, avoiding materialization of the full N×N attention matrix in HBM. At ~2,000 token context, the matrix is ~5M float16 elements (~10MB) — fits in L2 cache, so the HBM savings are minimal.

5. Evaluation Framework — Designed, Awaiting Implementation

Advisor feedback: a flat list of metrics lacks structure. The solution: a risk-based tiered framework with explicit priority ordering. Safety > Quality > Clinical Utility > Efficiency. Full details in results/evaluation_framework.html.

TierDimensionMetricsStatus
1 Safety (factuality vs source) QAFactEval (QA-based) + SummaC (NLI-based) + MedNER F1 (NER-based) Pending
2 Quality (relevance vs reference) ROUGE-1/2/L, BLEU-1..4, METEOR, BERTScore-F1 Done
3 Clinical Utility (LLM-as-Judge Jury) GPT-4o + DeepSeek-R1 + Claude 3.5 → PDSQI-9 rubric scoring Pending
4 Efficiency (deployment cost) Avg latency, total wall time, tokens/steps Collected

Why Three Methods in Tier 1

QAFactEval
QA-based. Generates questions from summary, answers from source. Detects entity substitutions and numerical errors. +14% over prior QA metrics on SummaC benchmark.
SummaC
NLI-based. Sentence-level entailment via RoBERTa-MNLI. Detects unsupported claims and logical contradictions. Complements QA-based: catches what QAFactEval misses.
MedNER F1
NER-based. Extracts medical entities (drugs, dosages, lab values) from source and summary. Computes precision/recall/F1. Domain-specific: catches errors general metrics miss.
Key citations: QAFactEval (Fabbri et al., NAACL 2022) proved QA+NLI ensemble > either alone. WeCheck (Wu et al., ACL 2023) confirmed joint weak supervision > single signal. Croxford et al. (npj Digital Medicine, 2025) validated LLM-as-Judge for clinical summarization (ICC=0.818). PoLL jury κ=0.906 (Bioengineering, 2026).

Key Observations

1. Fine-tuning consistently improves lexical overlap metrics. Both architectures show gains in ROUGE-2, BLEU, and METEOR after LoRA fine-tuning. LLaMA LoRA achieves the best scores on 5 of 6 lexical metrics. This is expected: fine-tuned models learn to mimic reference summary phrasing.

2. BERTScore drops after fine-tuning — the opposite of lexical metrics. LLaDA zero-shot leads BERTScore (0.2001) but drops to 0.1581 after fine-tuning. LLaMA shows a smaller drop (0.1711 → 0.1699). This suggests fine-tuned models may overfit to surface-level reference patterns while losing some semantic fidelity. This divergence between lexical and semantic metrics is a known phenomenon in NLG evaluation and highlights why single-metric evaluation is insufficient.

3. LLaDA zero-shot leads BERTScore. Despite being a diffusion model with no supervised fine-tuning on clinical text, LLaDA zero-shot achieves the highest BERTScore-F1 (0.2001). This may reflect differences in output diversity or phrasing patterns between diffusion and autoregressive generation, but the precise cause requires further analysis.

4. Overall, metrics are close across all 4 models. No model dominates across all metrics. The choice between architectures cannot be made on Quality alone — Safety (Tier 1) results will be decisive.
